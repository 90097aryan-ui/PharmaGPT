# PharmaGPT Security Review

**Scope:** `pharmagpt/` (Flask app, routes, db, services, auth), `scripts/`, `tests/`, top-level config/deployment files.
**Method:** Manual source review (no dynamic testing / no exploitation performed). Every finding below is cited to a real file:line found in the repository as of this review; no finding is speculative unless explicitly marked "not confirmed exploitable."
**Not in scope / not reviewed in depth:** `pharmagpt/static/js/*` (frontend), `pharmagpt/templates/*` content, third-party package internals (`check/`, `venv/` site-packages).

---

## Executive Summary

The application is mid-migration from a single-tenant SQLite datastore to a multi-tenant Postgres + Row-Level-Security architecture (documented in `docs/DATABASE_ARCHITECTURE.md` and `pharmagpt/config.py`'s `*_BACKEND` flags). **All backend flags default to `"sqlite"`**, meaning the SQLite tables are the actual, live read/write path for every domain today, including in any deployment that hasn't explicitly overridden the flags.

The single most important finding is that **the SQLite schema and every route built on it carry no `company_id`/tenant concept at all**. Authentication (Supabase Auth bearer token, verified via a global `before_request` hook) is solid and consistently applied — but authentication is not authorization. Once any user from any company is logged in, essentially every read/write/delete route (projects, documents, Knowledge Base, equipment, URS, QMS deviations/CAPAs/change-controls, risk assessments, DOCX exports) operates on a bare integer ID with no ownership check, so one tenant's authenticated user can read, modify, or delete another tenant's records simply by incrementing an ID. This is a systemic, cross-cutting Broken Access Control / IDOR issue, not a one-off bug in a single route — it is the architectural default of the currently-active data path. Given the docs describe this as a deliberate, tracked interim state of an in-progress migration, it may already be a known gap to the team, but as shipped/runnable code today it is a critical, live vulnerability if any real multi-tenant data exists in this SQLite database.

Aside from that, the codebase is otherwise careful: no SQL injection was found (every dynamic SQL string interpolates only hardcoded table/column literals or names drawn from fixed allow-lists, and all values are bound via `?` placeholders; the Postgres `*_repo.py` modules use the Supabase query builder exclusively, never raw SQL). No secrets are hardcoded in tracked source beyond one insecure-but-documented Flask `SECRET_KEY` fallback. File upload/download handling uses `secure_filename()` and DB-resolved (not client-supplied) paths, so no path traversal was found. No unsafe deserialization (`pickle`/`yaml.load`/`eval`/`exec`) exists anywhere in `pharmagpt/`.

## Findings by Category

### SQL Injection
No exploitable SQL injection found. See "Not a vulnerability" rows above — every dynamically-built query interpolates only hardcoded literals or statically allow-listed column names; all values are parameterized.

### Hardcoded Secrets / Credentials
One finding: `pharmagpt/config.py:15` insecure fallback `FLASK_SECRET_KEY`. No other hardcoded secrets found in tracked source; `.env` is correctly excluded from version control.

### Unsafe File Uploads
Extension allow-list (`pharmagpt/config.py:102`) is enforced (`pharmagpt/routes/docs.py:54`, `pharmagpt/routes/knowledge_base.py:116`); filenames are sanitized with `werkzeug.secure_filename()` (`pharmagpt/documents.py:95,136`) before being written to disk, preventing path traversal on upload. `MAX_CONTENT_LENGTH` is enforced globally via `app.config["MAX_CONTENT_LENGTH"] = MAX_FILE_SIZE` (`pharmagpt/app.py:48`). Only gap: no content/magic-byte validation beyond the extension (Low, see table).

### Unsafe File Downloads / Serving
All `send_file()` calls (`pharmagpt/routes/docs.py:130,149`, `pharmagpt/routes/knowledge_base.py:199,218`) build the path from a DB-stored `stored_filename` (itself produced by `secure_filename()` at upload time), not from client-supplied path input at request time — no path traversal found in the download/view routes themselves. However, these routes have no tenant/ownership check at all (see the Critical cross-tenant finding — any authenticated user can download any document by ID regardless of company).

### Broken Authentication
Authentication is centralized and consistently applied: `pharmagpt/auth/middleware.py`'s `before_request` hook (`register_auth_middleware`, wired in `pharmagpt/app.py:63`) gates every route except an explicit, small exempt list (`/`, `/auth/login`, `/health`, `/favicon.ico`, `/static/*` — `pharmagpt/auth/middleware.py:49-55`). Token verification is delegated to Supabase (`client.auth.get_user`, `pharmagpt/auth/context.py:49`), never performed against a locally-held JWT secret, and an inactive/deactivated account or a token with no matching `users` row is rejected (`pharmagpt/auth/context.py:66-73`). No route was found that is reachable without a valid Supabase session other than the documented exempt set. This category is in good shape.

### Broken Authorization / Privilege Escalation
See the Critical finding above — this is the primary result of this review.

### Missing Input Validation
Spot-checked several routes (`pharmagpt/routes/chat.py:40-52`, `pharmagpt/routes/projects.py:104-136`, `pharmagpt/routes/auth.py:44-49`) — basic presence/type checks exist (e.g., `email`/`password` required, `name` required, `project_id` required) but are generally shallow (no length limits, no schema validation library). No specific injection-relevant gap was found beyond what's already covered above; this is more a robustness/data-quality observation than a confirmed security vulnerability, so no table row was added for it beyond noting it here.

### Unsafe Deserialization
None found — no `pickle`, `yaml.load`, `eval`, or `exec` usage anywhere in `pharmagpt/`.

### CSRF / Cookie Flags / CORS
`SESSION_COOKIE_HTTPONLY = True`, `SESSION_COOKIE_SAMESITE = "Lax"`, `SESSION_COOKIE_SECURE = not FLASK_DEBUG` are all set (`pharmagpt/app.py:57-59`) — correct, environment-aware flags. No CORS configuration (e.g. `flask-cors`) was found anywhere in the codebase, meaning the default same-origin browser policy applies (no permissive `Access-Control-Allow-Origin: *` was set) — this is safe by omission for a same-origin SPA. No CSRF token framework exists, but risk appears mitigated by `SameSite=Lax` plus the bearer-token primary auth model (see Low finding above).

### Dependency Vulnerabilities
No specific CVE was confirmed against the pinned versions in `requirements.txt` during this manual read (see Informational row) — recommend an automated `pip-audit`/`safety` pass for authoritative results, which this review did not run.

## Notes on Confidence

- The cross-tenant authorization finding is the highest-confidence, highest-impact finding in this review: it was verified by reading the actual route handlers, the actual SQLite schema module, and confirming (via targeted grep across every `.py` file) that `company_id` never appears in any legacy `*_database.py` module.
- All "not a vulnerability" rows were reached by reading the actual call sites, not by pattern-matching alone — every dynamic-SQL f-string flagged by an initial broad grep was individually traced back to its allow-list or hardcoded-literal source.
- Frontend code (`pharmagpt/static/js/*`, template rendering in `pharmagpt/templates/*`) was not reviewed in this pass; if the extracted document text or any user-controlled field is ever rendered as raw HTML client-side, that would be a separate XSS surface this review did not check.
- Dependency-vulnerability coverage here is a manual read, not a CVE feed lookup, and should not be treated as authoritative — run `pip-audit` for that.
