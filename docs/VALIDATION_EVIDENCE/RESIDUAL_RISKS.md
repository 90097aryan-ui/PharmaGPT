# Validation Evidence — Residual Risks

Risks that remain even after every fix in this phase, and why they're accepted as residual rather than fixed now.

- **Operational dependency risk:** the single largest remaining blocker (A1/C1) is not a code defect this or any future coding session can fix in isolation — it requires someone with live Supabase SQL Editor access to re-apply three migration files and confirm the grants. Residual until that operational step happens, tracked explicitly rather than absorbed into a false sense of "code is done."
- **Business-rule risk from deliberately narrow scoping:** several Phase F fixes (C2 execution-only gate, C3 optional-linkage gate, terminal-immutability-only for H7) were deliberately scoped narrower than a maximal interpretation of the finding, specifically to avoid inventing unverified business rules or breaking legitimate existing workflows (proven by the 4 test failures this session hit and fixed by rescoping). This is the right call given the brief's explicit "do not add features" instruction, but it means the *narrowest* correct fix shipped, not the broadest conceivable one — a product owner with domain authority should confirm these scoping decisions match actual intended GxP process, particularly the CAPA/Change Control transition-graph gap (H7 residual).
- **Coverage risk:** new automated tests were added for the highest-value, most-argued-about fixes (RBAC, C2, C3, one audit-trail case) but not for every single one of the ~40+ call sites touched in Work Package 2. The rest are verified by direct code read, which is real verification but a weaker guarantee against future regression than a test.
- **Unverifiable-live risk:** the Postgres-side Users/Companies audit logging is coded correctly but cannot be proven to work against the live database from this session — it inherits the same live-access limitation as A1/C1.
- **Process risk:** this phase's live regression testing (`pytest`) mocks the Supabase client throughout, exactly as the original Enterprise Acceptance Test itself warned ("every one of these tests mocks the Supabase client... would not have caught anything in this report"). A green suite remains necessary but not sufficient proof for the Postgres/live-database-dependent findings.
