# Deployment Guide — Render

PharmaGPT deploys to [Render](https://render.com) as a `runtime: python` web service, defined in
[`render.yaml`](../render.yaml) at the repo root. This is the step-by-step guide; for the audit
that found the gaps this guide's checklist now closes, see
[`DEPLOYMENT_REVIEW.md`](../DEPLOYMENT_REVIEW.md) (historical — most items below are now fixed;
still useful for the reasoning behind each decision).

## 1. First-time setup

1. Create a new Render **Blueprint** from this repo (Render reads `render.yaml` automatically) or
   a new **Web Service** pointed at this repo with `runtime: python`.
2. Render provisions the `pharmagpt-data` persistent disk (5 GB, mounted at `/var/data`) declared
   in `render.yaml`. This holds the SQLite database, uploaded files, and AI-generated documents —
   all three would otherwise be wiped on every redeploy (Render's app filesystem is ephemeral;
   only the mounted disk survives restarts/redeploys).
3. Set the following secrets in the Render dashboard (declared with `sync: false` in `render.yaml`,
   so Render will not auto-populate them — you must set values by hand):
   - `GEMINI_API_KEY`
   - `SUPABASE_URL`
   - `SUPABASE_ANON_KEY`

   `FLASK_SECRET_KEY` is auto-generated by Render (`generateValue: true`) — no action needed.
4. Deploy. Render runs `pip install -r requirements.txt`, then starts gunicorn per
   `render.yaml`'s `startCommand`.

## 2. What `render.yaml` already handles

- **Health check** — `healthCheckPath: /health` (unauthenticated liveness endpoint,
  `pharmagpt/app.py`), used by Render for rollout gating and instance health.
- **Port binding** — `--bind 0.0.0.0:$PORT` in `startCommand`. Without this gunicorn defaults to
  `127.0.0.1:8000`, which Render's routing layer cannot reach.
- **Threaded workers** — `--worker-class=gthread --workers=2 --threads=4`. Gunicorn's default
  worker class (`sync`) silently ignores `--threads`; `gthread` is required for the concurrency
  `pharmagpt/database.py`'s SQLite WAL/`busy_timeout` setup is tuned for.
- **Persistent storage** — `DB_PATH`, `UPLOAD_FOLDER`, `GENERATED_DOCS_PATH` all point at
  subdirectories of the mounted disk (`/var/data/...`). All three fall back to paths inside the
  repo checkout when unset, which is correct for local dev but data-losing in production — always
  keep these three set in any persistent-filesystem deployment.
- **Fail-safe debug mode** — `FLASK_DEBUG=false` explicit in `render.yaml`, and the code-level
  default (`pharmagpt/config.py`) is also `"false"`, so an accidentally-removed env var fails safe
  rather than silently enabling the Werkzeug interactive debugger in production.
- **Migration flags left unset** — `DATABASE_BACKEND`/`PROJECTS_BACKEND`/`KB_BACKEND`/
  `EQUIPMENT_BACKEND`/`QMS_BACKEND` are intentionally not set, so every domain defaults to
  `sqlite`. See [`MIGRATION.md`](MIGRATION.md) before ever changing this.

## 3. Deployment checklist

- [x] `GEMINI_API_KEY`, `SUPABASE_URL`, `SUPABASE_ANON_KEY` set in the Render dashboard (secrets,
      not committed)
- [x] `FLASK_SECRET_KEY` auto-generated by Render
- [x] `healthCheckPath: /health` wired up
- [x] gunicorn bound to `0.0.0.0:$PORT`
- [x] gunicorn using `gthread` worker class so `--threads` actually takes effect
- [x] `DB_PATH`, `UPLOAD_FOLDER`, `GENERATED_DOCS_PATH` all point at the mounted persistent disk
- [x] `FLASK_DEBUG` fails safe (defaults to `false` at the code level, explicit `false` in
      `render.yaml`)
- [x] `pytest` and its transitive deps excluded from the production install
      (`requirements.txt` no longer includes `pytest`; it lives only in `requirements-dev.txt`)
- [x] Migration flags left at their `sqlite` default — cutover gate not yet satisfied, see
      [`MIGRATION.md`](MIGRATION.md)
- [ ] Load-test `--workers=2 --threads=4` on the `starter` plan against expected concurrent load —
      particularly SQLite write contention — before onboarding many concurrent customers
      (operational validation, not a code change)
- [ ] Confirm the 5 GB disk sizing is enough for actual document volume once real usage data
      exists; raise `sizeGB` in `render.yaml` if disk-usage alerts fire (operational, not urgent
      at pilot scale)
- [ ] Consider a CDN or platform static-asset offload for `pharmagpt/static/` once traffic volume
      makes gunicorn-served static assets a meaningful cost/latency factor (not urgent now)

## 4. Local dev vs production

| | Local dev | Render production |
|---|---|---|
| Entry point | `python pharmagpt/app.py` (Flask dev server) | gunicorn via `Procfile`/`render.yaml` `startCommand` |
| `FLASK_DEBUG` | `true` (set in `.env`) | `false` (fails safe even if unset) |
| `DB_PATH` / `UPLOAD_FOLDER` / `GENERATED_DOCS_PATH` | unset — default to paths inside the repo checkout | set to `/var/data/...` on the mounted disk |
| Secrets | `.env` (gitignored) | Render dashboard env vars |

See [`.env.example`](../.env.example) for the full environment variable reference.
